# Release 0.134.0 (2026-04-17)

Correctness overhaul across config generation, stack assembly, and global-config merge. Tightens contracts with hard invariants, introduces the ownership-based stack builder, and replaces the buggy "incoming wins" stack merge with a deep-additive merge that never silently removes or overwrites global state from project-context edits.

## Fixed

- **`mergeGlobalConfigs` silently dropped per-agent stack updates**. Previously, when incoming global partition and existing global config both contained the same agent, the merger either (a) skipped the incoming entry entirely ("existing wins"), losing new categories added for existing agents, or (b) overwrote with incoming ("incoming wins"), erasing existing categories not carried by incoming. Neither preserved the correct additive semantic. Replaced with **deep-additive merge per (agent, category, skill)** via new `additiveMergeStack` + `mergeAgentCategories` helpers in `src/cli/lib/installation/local-installer.ts`:
  - agent absent in existing → add from incoming
  - category absent in existing → add from incoming
  - skill id absent in existing → append from incoming
  - everything already present → keep existing as-is (including `preloaded` flag)
  - anything present only in `existing` → untouched

  Project-context edits to the global config are now purely additive — never remove or overwrite existing global state. Matches the commit `403df46` principle: "never modify global config from project-level operations". See `.ai-docs/agent-findings/2026-04-17-merge-global-configs-per-agent-update-loss.md`.

- **`generateProjectConfigFromSkills` silent scope fallbacks**. Callers that passed `selectedAgents` without a matching `skillConfigs`/`agentConfigs` previously got silent `"project"` scope defaults, masking bugs where per-skill scope information was missing. Now throws with a descriptive error naming the missing config. Scope lookups inside `buildAgentStack` use `getScopeOrThrow` (never defaults). See `.ai-docs/agent-findings/2026-04-15-silent-scope-fallbacks-in-config-generator.md`.

- **`matrix-provider.ts::getCategoryDomain` multi-tier fallback removed**. Previously fell back to `category.split("-")[0] + isDomain(prefix)` when `matrix.categories[category]` missed — a production fallback introduced to accommodate unit-test matrices, violating CLAUDE.md's "NEVER build multi-tier resolution fallbacks" rule. Removed; test fixtures corrected at the source. Dead `AgentName` and `isDomain` imports also removed. See `.ai-docs/agent-findings/2026-04-17-matrix-provider-multi-tier-fallback.md`.

## Changed

- **`buildAgentStack` rewritten** as an ownership-based builder (`src/cli/lib/configuration/config-generator.ts`). Each agent's stack is derived from `activeSkillsByCategory × selectedAgents × scope-compatibility` on every save. `preloaded` flag inherited from `existingStack` via `wasPreviouslyPreloaded`. `isScopeCompatible` enforces that project-scoped skills never land on global-scoped agents. See `.ai-docs/agent-findings/2026-04-15-stack-ownership-model-and-preloaded-preservation.md`.

- **`generateProjectConfigFromSkills` signature hardened** with explicit invariants. When `selectedAgents.length > 0`, `skillConfigs` AND `agentConfigs` must both be provided — missing either throws. `existingStack` threaded through for `preloaded` inheritance.

- **`config-merger.ts::mergeConfigs` stack policy simplified**. Stack is now the pure output of the mutator — `newConfig.stack` is authoritative when defined; `existingConfig.stack` only fills in when newConfig has none. Replaces the deep-merge-per-category policy that conflicted with the ownership builder's semantics.

- **`buildEjectConfig` threads `existingStack` from disk** into `generateProjectConfigFromSkills` so `preloaded` flags survive across edits. Stack-selected flow merges YAML-asserted `preloaded` flags via the same seam.

- **`mergeGlobalConfigs` exported** for unit testing. Single production caller: `writeScopedConfigs`.

- **`domains` and `selectedAgents` dirty detection** in `mergeGlobalConfigs` now uses `isDeepEqual` (instead of `length >`) — catches same-length-different-members cases for full correctness.

## Added

- **New E2E tests for stack and scope round-trip correctness:**
  - `e2e/lifecycle/preloaded-preservation.e2e.test.ts` — guards that stack YAML author-asserted `preloaded` flags survive re-edits via `wasPreviouslyPreloaded` inheritance
  - `e2e/lifecycle/scope-toggle-roundtrip.e2e.test.ts` — G→P toggle persists through edit re-open; config + filesystem agree
  - `e2e/lifecycle/scope-toggle-config-snapshot.e2e.test.ts` — config.ts byte-level stability across scope toggles
  - `e2e/lifecycle/scope-toggle-agent-content.e2e.test.ts` — compiled agent .md files reflect the correct skill set after scope changes
  - `e2e/lifecycle/scope-toggle-combined.e2e.test.ts` — multi-skill, multi-agent scope-toggle interactions

- **8 new unit tests** in `local-installer.test.ts::describe("mergeGlobalConfigs")`, covering:
  - New category added to existing agent's stack (additive)
  - New skill appended to existing category under existing agent (additive)
  - New agent added (incoming-only)
  - Existing agent preserved when incoming omits it
  - Identical merge → `changed: false`
  - Existing skills preserved when incoming omits one (tombstone preservation)
  - Existing `preloaded` flag preserved when incoming has a different value
  - Existing category preserved when incoming omits it for an existing agent
  - Input non-mutation: both `existing` and `incoming` snapshot-equal after call; merged stack arrays unaliased from both inputs
