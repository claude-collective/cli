# Release 0.132.0 (2026-04-15)

Eliminates a one-frame flash of the stack-selection screen on `cc edit` startup, and fixes an edit-mode escape-navigation footgun where two escape presses could walk the user back to a stack-selection screen they never chose.

## Fixed

- **First-frame flash on `cc edit`**. `Wizard` subscribed to the Zustand store via `useWizardStore()` before `useWizardInitialization()` ran, so the first-render snapshot captured `step: "stack"` (the module-level default). The hook's render-phase `setState({ step: "build", ... })` scheduled a re-render after Ink had already committed the first frame to stdout — the stack-selection screen was briefly visible. Initialization now runs imperatively via `hydrateWizardStore(options)` called BEFORE `render(<Wizard>)`. The store enters the correct state before React subscribes; the first frame paints the intended step.

- **Edit-mode escape walked users back to stack-selection**. Previously, two escape presses from the build step walked through fabricated history entries (built by the old hook's `setStep()` walk from `"stack"` through each intermediate step) and landed on the stack-selection screen. With hydrate jumping directly to `initialStep` with `history: []`, combined with `goBack` now no-oping on empty history (instead of falling back to `"stack"`), users stay on their starting step.

## Changed

- **`useWizardInitialization` deleted.** Its logic moved into `hydrateWizardStore(options): void` co-located with the store in `src/cli/stores/wizard-store.ts`. `createInitialState(overrides?)` is now exported and accepts optional overrides.

- **`Wizard` component props slimmed down.** Five init-only props removed: `initialStep`, `initialDomains`, `installedSkillConfigs`, `installedAgentConfigs`, `isEditingFromGlobalScope`. Their flow is owned by `hydrateWizardStore`. Two props kept because they have render-time consumers:
  - `initialAgents` — launch-time closure in `StepSources.onContinue` gating `preselectAgentsFromDomains`. Preserves the semantic that if the user started with agents selected, continuing past sources doesn't silently re-populate domain defaults when they've deselected everything.
  - `installedSkillIds` — drives the build-step dimmed-checkmark indicator via `useBuildStepProps`. Value comes from `edit.tsx` as `(plugin-discovered ∪ projectConfig.skills) − excluded`, which is not equivalent to `projectConfig.skills` and cannot be correctly re-derived from the store alone.

- **`goBack` no-op on empty history.** Previously fell back to `step: "stack"`, which masked the intent "there's nowhere to go back to." Now returns unchanged state, aligned with CLAUDE.md's "no silent fallbacks" principle. Non-null assertion on `history.pop()` after the length check matches the pattern used elsewhere in the store.

## Added

- **`hydrateWizardStore(options: HydrateOptions): void`** exported from `src/cli/stores/wizard-store.ts`. Resets to `createInitialState()`, then applies option-driven overrides in the exact sequence previously handled by `useWizardInitialization`: `isInitMode` → `populateFromSkillIds` (edit flow only) → step jump → `selectedDomains` override → `selectedAgents` override → `agentConfigs` override → installed-config snapshots → global-scope flag → `globalPreselections`/`globalAgentPreselections` (init flow only). Call from `edit.tsx` and `init.tsx` before `render(<Wizard>)`.

- **First-frame regression test** in `wizard-flow.integration.test.tsx`. Uses `ink-testing-library`'s `frames` array to assert that `frames[0]` does not contain stack-selection content when hydrated with `initialStep: "build"`. Catches the exact flash this release fixes.

- **Six `hydrateWizardStore` unit tests** in `wizard-store.test.ts` covering edit-flow state, init-flow defaults, `installedSkillConfigs` diff snapshot, `isEditingFromGlobalScope`, fresh-state reset before hydration, and init-flow `globalPreselections`.

- **Two `goBack` empty-history tests** in `wizard-store.test.ts` pinning down the no-op behavior from both `setStep("build") → goBack() → goBack()` and direct `setState({ step: "build", history: [] })` entry paths.

- **Three "restored render-time props" regression tests** in `wizard-flow.integration.test.tsx`:
  1. `installedSkillIds` prop forwards to `buildCategoriesForDomain` (not re-derived from store) — prevents a future refactor from silently narrowing the data set.
  2. `initialAgents` prop present → no preselect on `onContinue` (launch-time closure) — prevents a refactor from flipping to a click-time `store.selectedAgents.length` read that would silently re-populate user-deselected agents.
  3. `initialAgents` prop absent → preselect fires — ensures the gate inversion is correct for init mode.

## Removed

- **`src/cli/components/hooks/use-wizard-initialization.ts`** deleted. Logic consolidated into `hydrateWizardStore`.

## Investigation — logged as D-215

- **Proposed** (ready for dev): emit singular for exclusive categories, array for multi-select; drop `preloaded: false` emission (it's the default, bare string round-trips identically); drop redundant `selectedAgents` field from `ProjectConfig` (derivable from `agents: AgentScopeConfig[]`); rename `domains` → `selectedDomains` to match wizard store field name.
- **Rejected**: dropping domain prefix from category keys (5-way `framework` collision across web/api/cli/mobile/desktop; `tooling` collides within `web-developer` which references both `web-tooling` and `shared-tooling`; `populateFromStack` relies on direct `matrix.categories[key]` lookup and would silently skip entries).
- **Rejected**: skill slugs instead of IDs (slugs not structurally unique).
- **No migration shim needed**: loader schema already accepts all target shapes via `z.union([element, z.array(element)])`; `edit` rewrites the full config on every run so existing configs auto-upgrade implicitly.
