# D-37: Dual-Installation Resolution (Global + Project)

**Status:** Design
**Includes:** D-74 (per-agent scope — implemented as Phase 4)
**Date:** 2026-03-06
**Supersedes:** `D-37-merge-installs.md` (outdated — assumed separate `--global` flag)

---

## Overview

A single `agentsinc init` run creates two separate installations — global (`~/`) and project (`./`) — based on per-item scope choices. Each installation has its own config files. The project config imports from global and extends it, so the CLI and compiler see a unified view via standard TypeScript imports.

---

## How Scope Works Today (0.58.0)

### Plugin mode — already routes correctly

- `scope: "global"` → `claude plugin install <ref> --scope user` → Claude's user-level registry, available in all projects
- `scope: "project"` → `claude plugin install <ref> --scope project` → project-level registry, this project only

Claude Code handles discovery — if a plugin is installed with `--scope user`, Claude picks it up everywhere. No compiler changes needed for plugin-mode global skills.

### Local mode — does NOT yet route by scope

All local skills go to `{projectDir}/.claude/skills/` regardless of scope. This is the gap.

### Agents — no scope concept yet

Agents are `AgentName[]` in config. Always compiled to `{projectDir}/.claude/agents/`. Phase 4 of this doc adds per-agent scope.

## Target Architecture

### Directory layout

```
~/.claude-src/                      # Global installation config
  config.ts                         # Global skills + agents config
  config-types.ts                   # Global type unions (SkillId, AgentName, etc.)

~/.claude/                          # Global installation output
  skills/                           # Global local-mode skills
    meta-methodology-investigation/ # SKILL.md, metadata.yaml
    meta-methodology-anti-over-eng/ # ...
  agents/                           # Global compiled agents
    documenter.md
    researcher.md
  plugins/                          # Global plugin-mode skills (managed by Claude)

<project>/.claude-src/              # Project installation config
  config.ts                         # Project skills + agents, imports from global
  config-types.ts                   # Project types, imports and extends global

<project>/.claude/                  # Project installation output
  skills/                           # Project local-mode skills
    web-framework-react/
    web-state-zustand/
  agents/                           # Project compiled agents
    web-developer.md
  plugins/                          # Project plugin-mode skills (managed by Claude)
```

### Config relationship

The project `config.ts` imports from the global config and extends it:

```typescript
// <project>/.claude-src/config.ts  (generated)
import globalConfig from "/home/vince/.claude-src/config";
import type { ProjectConfig } from "./config-types";

export default {
  ...globalConfig,
  name: "my-project",
  skills: [
    ...globalConfig.skills,
    { id: "web-framework-react", scope: "project", source: "local" },
    { id: "web-state-zustand", scope: "project", source: "local" },
  ],
  agents: [...globalConfig.agents, { name: "web-developer", scope: "project" }],
  domains: [...(globalConfig.domains ?? []), "web"],
} satisfies ProjectConfig;
```

The project `config-types.ts` imports global types and extends:

```typescript
// <project>/.claude-src/config-types.ts  (generated)
import type {
  SkillId as GlobalSkillId,
  AgentName as GlobalAgentName,
  Domain as GlobalDomain,
} from "/home/vince/.claude-src/config-types";

export type SkillId =
  | GlobalSkillId
  // Project
  | "web-framework-react"
  | "web-state-zustand";

export type AgentName =
  | GlobalAgentName
  // Project
  | "web-developer";

export type Domain =
  | GlobalDomain
  // Project
  | "web";

// ... rest of type definitions
```

### First init — auto-create blank global

The first time `agentsinc init` runs and no global installation exists, it creates a blank global config:

```typescript
// ~/.claude-src/config.ts  (generated)
import type { ProjectConfig } from "./config-types";

export default {
  name: "global",
  skills: [],
  agents: [],
  domains: [],
} satisfies ProjectConfig;
```

```typescript
// ~/.claude-src/config-types.ts  (generated)
// AUTO-GENERATED by agentsinc — DO NOT EDIT

export type SkillId = never;
export type AgentName = never;
export type Domain = never;
// ... rest with `never` unions
```

This ensures the project config can always import from global — no conditional logic needed.

## Implementation Phases

### Phase 1: Local installer path routing

**Goal:** `scope: "global"` local skills install to `~/.claude/skills/`, `scope: "project"` to `.claude/skills/`.

**Files:**

- `src/cli/lib/installation/local-installer.ts` — `resolveInstallPaths()` takes scope, routes to `os.homedir()` or `projectDir`
- `src/cli/commands/init.tsx` — pass scope when installing local skills
- `src/cli/commands/edit.tsx` — same

**Change:** In `resolveInstallPaths()` (or a new scope-aware wrapper), when `scope === "global"`, use `os.homedir()` as the base directory instead of `projectDir`.

### Phase 2: Config splitting on save

**Goal:** Wizard result gets split by scope. Global items → `~/.claude-src/config.ts`. Project items → `.claude-src/config.ts` with import from global.

**Files:**

- `src/cli/lib/configuration/config-generator.ts` — split `generateProjectConfigFromSkills()` output by scope
- `src/cli/lib/configuration/config-writer.ts` — write to two locations, project config includes import
- `src/cli/lib/configuration/config-types-writer.ts` — project types import and extend global types
- `src/cli/commands/init.tsx` — auto-create blank global on first init

**New function:** `splitConfigByScope(config: ProjectConfig) → { global: ProjectConfig, project: ProjectConfig }` — partitions skills, agents, domains by scope.

**Config writer changes:** When writing project config, generate the import preamble from global. The absolute path to `~/.claude-src/config` is resolved at generation time via `os.homedir()`.

### Phase 3: Compiler resolves from both directories

**Goal:** `agentsinc compile` finds skills from both global and project local skill directories.

**Files:**

- `src/cli/commands/compile.ts` — `discoverAllSkills()` scans both `~/.claude/skills/` and `.claude/skills/`

**Change (~5 lines):**

```typescript
// In discoverAllSkills():
const globalLocalSkills = await loadSkillsFromDir(
  path.join(os.homedir(), LOCAL_SKILLS_PATH),
  LOCAL_SKILLS_PATH,
);
const allSkills = mergeSkills(pluginSkills, globalLocalSkills, localSkills);
// Local project skills take precedence over global (later sources win)
```

**Agent output routing:** When writing compiled agent `.md` files, check the agent's scope:

- `scope: "global"` → `path.join(os.homedir(), CLAUDE_DIR, "agents", name + ".md")`
- `scope: "project"` → `path.join(projectDir, CLAUDE_DIR, "agents", name + ".md")`

**Cross-scope skill filtering:** When building the skill list for an agent, global agents only see global skills. Project agents see everything (see D7).

### Phase 4: Per-agent scope (D-74)

**Goal:** Agents get the same scope treatment as skills.

**Files:**

- `src/cli/types/config.ts` — `AgentConfig = { name: AgentName, scope: "project" | "global" }`
- `src/cli/stores/wizard-store.ts` — `agentConfigs`, `toggleAgentScope`, `focusedAgentId`
- `src/cli/components/wizard/step-agents.tsx` — S key toggle, [P]/[G] badge
- `src/cli/components/wizard/step-confirm.tsx` — agent scope summary
- `src/cli/lib/stacks/stack-plugin-compiler.ts` — scope-aware agent output path

### Phase 5: Edit flow

**Goal:** `agentsinc edit` loads the unified config (project imports global), shows merged view, splits on save.

**How it works:** The project `config.ts` already includes global items via the TypeScript import. When the CLI loads the config (via jiti), it gets the full merged object. The wizard shows everything with scope badges ([P]/[G]). On save, items are split by scope and written to the correct config file.

**Files:**

- `src/cli/commands/edit.tsx` — load config (already merged via import), split on save
- `src/cli/components/hooks/use-wizard-initialization.ts` — populate wizard with scope-tagged items

**Scope change during edit:** If a user toggles a skill from [P] to [G] (or vice versa):

1. The skill's `SkillConfig.scope` changes
2. On save, the skill moves between configs
3. For local mode: skill files migrate between `.claude/skills/` and `~/.claude/skills/`
4. For plugin mode: `claude plugin uninstall --scope project` + `claude plugin install --scope user` (or reverse)

## Design Decisions

1. **Auto-create blank global on first init** — eliminates conditional import logic. Project config always imports from `~/.claude-src/config`.

2. **Absolute home path in imports** — `import from "/home/user/.claude-src/config"`. Works because `.claude-src/` is gitignored — the import is machine-local. The path is resolved via `os.homedir()` at generation time.

3. **Both local and plugin global skills supported** — global scope works for both `source: "local"` (files in `~/.claude/skills/`) and `source: "marketplace"` (plugins via `claude plugin install --scope user`). The compiler scans both directories.

4. **Project config is the entry point (in project context)** — when running from a project directory, the CLI loads the project config, which includes global via import. No separate merge step at runtime. When running from `~/`, the CLI loads the global config directly.

5. **Scope migration during edit** — changing a skill's scope triggers file migration (local mode) or plugin re-registration (plugin mode) between global and project directories.

## Command-by-Command Impact

Beyond the core phases above, these commands need scope awareness:

| Command     | Change needed                                                                              |
| ----------- | ------------------------------------------------------------------------------------------ |
| `compile`   | Phase 3 — scan both `~/.claude/skills/` and `.claude/skills/`, route agent output by scope |
| `init`      | Phase 2 — split config by scope on save, auto-create blank global                          |
| `edit`      | Phase 5 — load merged config, split on save, migrate files on scope change                 |
| `list`      | Show scope badge ([P]/[G]) next to each skill/agent                                        |
| `doctor`    | Validate both global and project installations, check global config exists                 |
| `new skill` | Route created skill to correct directory based on `--scope` flag or default                |
| `eject`     | Eject from correct directory (global skills from `~/`, project from `./`)                  |
| `uninstall` | Remove from correct directory/registry based on skill's scope                              |
| `import`    | Route imported skill to correct directory based on scope                                   |

### Installation type changes

`Installation` in `installation.ts` needs a `globalDir` field (or derives it from `os.homedir()`). The `detectInstallation()` function should check both locations.

### Deduplication

If the same skill ID exists in both global and project, project wins. The compiler already uses a merge pattern where later sources override earlier ones — global skills are loaded first, project skills second.

## Implementation Notes

### Config writer is string templating

The config writer (`config-writer.ts`) generates TypeScript source as text via string concatenation. The import preamble for the project config is just another template string:

```typescript
const importLine = `import globalConfig from "${homedir}/.claude-src/config";`;
const configBody = `export default { ...globalConfig, ... } satisfies ProjectConfig;`;
```

Same for `config-types-writer.ts` — the global type imports are templated as text. No AST manipulation or `JSON.stringify` changes needed.

## Resolved Decisions

### D1: Deleted global config

When the project config can't import from `~/.claude-src/config`, jiti throws `MODULE_NOT_FOUND`. The config loader catches this, checks if the missing path contains `.claude-src/config`, and shows: "Global installation not found — run `agentsinc init` to recreate it."

On `agentsinc uninstall` in the global directory: warn that files will be emptied but `~/.claude-src/` directory and types file will remain (so existing project imports don't crash at the filesystem level — they import an empty config).

### D2: Removing global items can break other projects

Not a concern. TypeScript will catch it. If a user removes a global skill that project agents depend on, the project config's type check will fail. No defensive programming needed — this is pre-1.0.

### D3: Init vs edit — scope is determined by working directory

- `init` in a project dir → creates project installation + scaffolds blank global if none exists
- Project already has installation → must use `edit` (existing behavior)
- `init` in `~/` (global dir) → creates global installation directly
- Global already has installation → must use `edit`
- `edit` always operates on the installation in the current directory
- The CLI only reaches out to global from within a project context (via the TypeScript import in the project config)

### D4: Default scope is project

New items default to `scope: "project"`. User presses S to toggle to `"global"` (opt-in).

### D5: Rename "global" → "user" scope (LAST)

The current codebase uses `"global"` but Claude Code's plugin system uses `--scope user`. Rename to `"user"` for consistency. **Do this as the final mechanical refactor** after all feature work is done — simple find-and-replace, don't mix with feature changes.

### D6: Compile always rebuilds both

`agentsinc compile` rebuilds all agents regardless of scope. Global agents write to `~/.claude/agents/`, project agents to `./.claude/agents/`. No `--global` flag needed.

### D7: Cross-scope reference rule

**Project agents can reference global skills. Global agents cannot reference project skills.**

Enforced at compile time with a simple filter:

```typescript
const availableSkills =
  agent.scope === "global" ? allSkills.filter((s) => s.scope === "global") : allSkills; // project agents see everything
```

This is naturally consistent with the config structure: global `config.ts` only lists global skills, so a global agent's stacks can only reference global skills. The filter is a safety net (~2 lines). (Note: `"global"` becomes `"user"` after D5 rename.)

### D8: Shared projects (team repos)

`.claude-src/` is gitignored — each developer runs `init` independently. If someone commits `.claude-src/`, the absolute home path import breaks. The CLI should warn if `.claude-src/` is not in `.gitignore`.

### D9: Global items in project wizard — read-only but promotable

When running `init` or `edit` from a project directory:

- **Existing global items** (already in `~/.claude-src/config.ts`) show as pre-selected with [G] badge and are **disabled** — the user cannot toggle or remove them.
- **Project items can be promoted to global** — pressing S on a [P] item toggles it to [G], which moves it to the global config on save.
- To remove or demote existing global items, run `edit` from `~/`.

### D10: Init from `~/` is supported

`init` in the home directory creates a global-only installation (no import preamble, no project config). One if-statement: if `cwd === os.homedir()`, write config directly to `~/.claude-src/` without the global import.

### D11: Cross-scope resolution is just config

Global agents reference global skills because the global `config.ts` only lists global skills. No special filtering mechanism at compile time beyond the safety net in D7. The config is already correctly scoped because it was split on save.

### D12: Concurrent global modification is not a concern

Two edit sessions in different projects could both write to global config. Last write wins. Unlikely in practice (single user, sequential commands). No mitigation for pre-1.0.
