## 1. Executive Summary

The CLI uses a **hybrid loading strategy** optimized for startup performance:

- **Pre-computed artifact**: BUILT_IN_MATRIX generated at build-time, shipped as ~9KB JSON literal
- **Lazy initialization**: Module-scoped `matrix` variable starts with BUILT_IN_MATRIX, replaced on-demand
- **Deferred merging**: Expensive work (YAML parsing, relationship resolution) happens per-command
- **Custom skills at runtime**: Local/project skills merged on-demand, not pre-built

**Key insight**: Commands that don't use matrix (--help, --version) still pay ~200ms cost for BUILT_IN_MATRIX deserialization due to oclif's eager command loading pattern.

### What's Included vs Excluded

**Included** (pre-computed):

- Built-in marketplace skills (~200 skills)
- Default categories from `defaultCategories` (30+ categories)
- Default relationships from `defaultRules` (conflicts, requires, alternatives, etc.)
- All resolved skill relationships (no lazy computation)
- Slug bidirectional map
- Suggested stacks (converted to ResolvedStack)

**NOT included** (computed at runtime):

- Local/custom skills (`~/.claude/skills/`, `<project>/.claude/skills/`)
- Extra source skills (per project configuration)
- Plugin skills (from settings.json)
- Multi-source availability metadata (added by loadSkillsFromAllSources)

### Performance Impact

**Deserialization cost**: ~200ms on module import (first access only)

- Node.js parses JSON and creates object graph
- Subsequent accesses are in-memory (fast)
- V8 inlines constant objects for optimization

**Runtime cost**: Negligible (~0.1ms for assignment)

### Build Command

- AGENT_NAMES array

**Build-time I/O**:

- Reads all `metadata.yaml` files from `../../skills/src/skills/`
- Reads SKILL.md frontmatter for skill names/descriptions

### Phase 2: Full Matrix Computation (276-353)

**Function**: `generatePhase2(skills, agentEntries, skillIdSet, outDir)`

**Output file**: `src/cli/types/generated/matrix.ts`

**Process**:

1. Call `mergeMatrixWithSkills(defaultCategories, defaultRules.relationships, skills)`

3. Serialize entire MergedSkillsMatrix to JSON
4. Write as TypeScript const export

**Build-time I/O**:

- Reads skill metadata extracted in Phase 1
- Reads skill-categories.ts from sibling skills repo (if exists)
- Reads skill-rules.ts from sibling skills repo
- Writes ~9KB TypeScript file with pre-stringified BUILT_IN_MATRIX

**Key optimization**: Uses `JSON.stringify()` to pre-compute the object representation once, storing as a literal in source code.

### Function: mergeMatrixWithSkills()

### Step-by-Step Process

#### 5.1: Build Slug Map (53-71)

Creates bidirectional slug ↔ ID mapping:

```typescript
function buildSlugMap(skills: ExtractedSkillMetadata[]): SkillSlugMap {
  const slugToId: Partial<Record<SkillSlug, SkillId>> = {};
  const idToSlug: Partial<Record<SkillId, SkillSlug>> = {};

  for (const skill of skills) {
    slugToId[skill.slug] = skill.id; // e.g. "react" → "web-framework-react"
    idToSlug[skill.id] = skill.slug; // reverse lookup
  }

  return { slugToId, idToSlug };
}
```

**Cost**: O(n) where n = number of skills

#### 5.2: Resolve Each Skill (108-111)

For each skill, call `buildResolvedSkill()` to resolve relationships:

```typescript
for (const skill of skills) {
  const resolved = buildResolvedSkill(skill, categories, relationships, slugMap);
  resolvedSkills[skill.id] = resolved;
}
```

**Cost**: O(n \* m) where:

- n = number of skills (~200)
- m = average relationship rules per skill (~45)

#### 5.3: Resolve Relationships (150-237)

For each skill, traverse all relationship rules and collect matching references:

```typescript
function resolveRelationships(
  skillId: SkillId,
  relationships: RelationshipDefinitions,
  resolve: ResolveId,
): ResolvedRelationships {
  // For each relationship type:
  // 1. Iterate through all rules
  // 2. Filter rules by skillId (does this skill participate?)
  // 3. For each matching rule, resolve slug references to SkillIds
  // 4. Deduplicate and collect results
}
```

**Relationship types** (in detail):

**Conflicts** (lines 162-172): Symmetric groups

- Find rules where skillId is mentioned
- Collect all other members of the group
- Store with conflict reason

**Discourages** (lines 175-187): Symmetric groups

- Find rules where skillId is mentioned
- Collect other members (bidirectional relationship)

**Requirements** (lines 203-215): Directional

- Find rules where `rule.skill` matches skillId (identifies dependent)
- Collect `rule.needs` as dependencies
- Supports `needsAny` (OR) or all (AND) semantics

**CompatibleWith** (lines 190-200): Symmetric groups

- Find groups where skillId appears
- Collect all co-members (framework compatibility groups)

**Alternatives** (lines 218-228): Symmetric groups

- Find groups where skillId appears
- Collect alternatives with purpose

**Cost per skill**: O(rules _ avg_group_size _ slug_lookups)

- ~45 rules per skill
- ~3 skills per rule group average
- ~1 slug lookup per resolution attempt

**Total for BUILT_IN_MATRIX**: ~27,000 string comparisons (pre-computed)

#### 5.4: Auto-Synthesize Categories (114-123)

For skills referencing undefined categories, create them:

```typescript
for (const skill of skills) {
  if (skill.category === "local") continue; // pseudo-category
  if (!synthesizedCategories[skill.category]) {
    const synthesized = synthesizeCategory(skill.category, skill.domain);

```

**synthesizeCategory()** (29-46):

```typescript
export function synthesizeCategory(category: Category, domain: Domain): CategoryDefinition {
  const displayName = category
    .split("-")
    .map((word) => word.charAt(0).toUpperCase() + word.slice(1))
    .join(" ");

  return {
    id: category,
    displayName, // e.g. "web-framework" → "Web Framework"
    description: `Auto-generated category for ${category}`,
    domain,
    exclusive: false,
    required: false,
    order: 999,
  };
}
```

**Cost**: O(n) where n = unique undefined categories (usually 0)

#### 5.5: Return Merged Matrix (125-134)

```typescript
const merged: MergedSkillsMatrix = {
  version: "1.0.0",
  categories: synthesizedCategories,
  skills: resolvedSkills,
  suggestedStacks: [], // populated later

```

### Properties of mergeMatrixWithSkills()

- **Pure function**: No file I/O, no side effects
- **Deterministic**: Same input always produces same output
- **Safe to pre-compute**: Output is static until skills/relationships change
- **Used twice**: Once during build (phase 2), once per custom source load

## 7. Custom/Local Skills Merging

### Function: mergeLocalSkillsIntoMatrix()

**File**: `src/cli/lib/loading/source-loader.ts:364-425`

**Process**:

```typescript
export function mergeLocalSkillsIntoMatrix(
  matrix: MergedSkillsMatrix,
  localResult: LocalSkillDiscoveryResult,
): MergedSkillsMatrix {
  for (const metadata of localResult.skills) {
    const existingSkill = matrix.skills[metadata.id];

    // If overwriting remote skill, inherit category/slug/displayName
    const category = existingSkill?.category ?? metadata.category;
    const slug = existingSkill?.slug ?? metadata.slug;
    const displayName = existingSkill?.displayName ?? metadata.displayName;

    // Create ResolvedSkill
    const resolvedSkill: ResolvedSkill = {
      id: metadata.id,
      slug,
      displayName,
      description: metadata.description,
      usageGuidance: metadata.usageGuidance,
      category,
      author: LOCAL_DEFAULTS.AUTHOR,

      // Inherit relationships from existing skill (or empty)
      conflictsWith: existingSkill?.conflictsWith ?? [],
      isRecommended: existingSkill?.isRecommended ?? false,
      recommendedReason: existingSkill?.recommendedReason,
      requires: existingSkill?.requires ?? [],
      alternatives: existingSkill?.alternatives ?? [],
      discourages: existingSkill?.discourages ?? [],
      compatibleWith: existingSkill?.compatibleWith ?? [],

      path: metadata.path,
      local: true,
      localPath: metadata.localPath,
      custom: metadata.custom,
    };

    matrix.skills[metadata.id] = resolvedSkill;

    // Ensure category exists
    if (category !== "local" && !matrix.categories[category] && metadata.domain) {
      matrix.categories[category] = {
        id: category,
        displayName: category,
        description: `Local skill category`,
        domain: metadata.domain,
        exclusive: false,
        required: false,
        order: 0,
      };
    }
  }

  return matrix;
}
```

### Key Properties

- **Additive**: Adds new skills, doesn't remove existing ones
- **Override-safe**: Preserves category/slug/displayName from existing skill
- **Relationship-preserving**: Local skills inherit relationships from base matrix
- **Mutates matrix in place**: Returns same matrix object with added skills
- **Custom skill support**: Marks skills with `custom: true` flag

### Eager Load Chain

When oclif loads a command:

1. **oclif/core** discovers command class
2. Imports command file (e.g., `src/cli/commands/init.tsx`)
3. Command imports utility modules (e.g., operations, components)
4. Utilities import `matrix-provider.ts`
5. `matrix-provider.ts` imports `BUILT_IN_MATRIX`

**Result**: All commands that import matrix-using modules pay ~200ms cost at startup.

**--version, --help**: These are handled by oclif before command classes load, but:

- oclif's strategy pattern still loads all command class definitions
- Commands import matrix-using utilities at top level
- Cannot avoid ~200ms cost with current architecture

## 10. Work Distribution (Eager vs Per-Command)

### Eager (on CLI import, before command.run())

1. **matrix-provider.ts import** (~0.1ms)
   - Assignment `let matrix = BUILT_IN_MATRIX`
   - Variable now references pre-computed object

2. **BUILT_IN_MATRIX deserialization** (~200ms)
   - First access to BUILT_IN_MATRIX
   - Node.js parses ~9KB JSON from source
   - Creates object graph in memory

3. **All command imports** (~100ms for init/edit/doctor/etc.)
   - oclif loads all command classes via strategy pattern
   - Each imports matrix-using modules
   - No computation, just imports

**Total eager cost**: ~300ms (unavoidable for interactive commands)

### Per-Command (when command.run() executes)

#### For `agentsinc init` / `agentsinc edit`:

1. **loadSource() operation** (~1000ms or more)
   - Calls `loadSkillsMatrixFromSource()`

   **For DEFAULT_SOURCE** (typical case):
   - `resolveSource()`: ~10ms
   - `fetchFromSource(DEFAULT_SOURCE)`: ~50ms cached, ~2s first time
   - Create shallow copies of BUILT_IN_MATRIX: ~0.1ms

## 14. Anti-Patterns Found

### Pattern 1: Shallow Spread on Mutable Objects

**File**: `src/cli/lib/loading/source-loader.ts:90-95`

```typescript
result = {
  matrix: {
    ...BUILT_IN_MATRIX, // ← shallow spread
    skills: { ...BUILT_IN_MATRIX.skills }, // ← shallow spread
    categories: { ...BUILT_IN_MATRIX.categories }, // ← shallow spread
    suggestedStacks: [...BUILT_IN_MATRIX.suggestedStacks],
  },
  // ...
};
```

**Problem**:

- Top-level spread creates new object (good for avoiding mutations of BUILT_IN_MATRIX)
- But nested objects (skills, categories) are still references to same objects
- If code mutates skill properties, it mutates BUILT_IN_MATRIX's skills
- The pattern suggests intent to avoid mutations but is incomplete

**Impact**: Low (code doesn't appear to mutate skill properties afterward)

**Better alternatives**:

Option 1: Use directly if no mutations:

```typescript
result = { matrix: BUILT_IN_MATRIX, ... };
```

Option 2: Deep clone if mutations needed:

```typescript
const cloned = structuredClone(BUILT_IN_MATRIX);
result = { matrix: cloned, ... };
```

Option 3: Freeze to prevent mutations:

```typescript
Object.freeze(BUILT_IN_MATRIX);
result = { matrix: BUILT_IN_MATRIX, ... };
```

## 15. Optimization Opportunities

### Ranked by Impact and Feasibility

#### 1. Lazy BUILT_IN_MATRIX Import (saves ~200ms, HIGH feasibility)

**Current**: matrix-provider.ts eagerly imports BUILT_IN_MATRIX

```typescript
import { BUILT_IN_MATRIX } from "../../types/generated/matrix";
export let matrix: MergedSkillsMatrix = BUILT_IN_MATRIX;
```

**Problem**: All modules that import matrix-provider.ts must deserialize BUILT_IN_MATRIX, even if they don't use it (e.g., --help commands)

**Solution**: Lazy-load on first access

```typescript
let _matrix: MergedSkillsMatrix | undefined;

function getDefaultMatrix() {
  if (!_matrix) {
    const { BUILT_IN_MATRIX } = await import("../../types/generated/matrix.js");
    _matrix = BUILT_IN_MATRIX;
  }
  return _matrix;
}

export function getMatrix(): MergedSkillsMatrix {
  return _matrix || getDefaultMatrix();
}
```

**Impact**: Saves ~200ms for --help/--version commands

**Feasibility**: Medium

- Requires changing all import sites
- Async initialization adds complexity
- oclif command loading still happens

#### 2. Parallel Skill Extraction (saves ~200ms on custom sources, MEDIUM feasibility)

**Current**: extractAllSkills() uses sequential globbing and file reads

```typescript
const metadataFiles = await glob(`**/${STANDARD_FILES.METADATA_YAML}`, skillsDir);
for (const metadataFile of metadataFiles) {
  // ... read and parse sequentially
}
```

**Solution**: Extract in parallel with Promise.all()

```typescript
const metadataFiles = await glob(`**/${STANDARD_FILES.METADATA_YAML}`, skillsDir);
const skills = await Promise.all(
  metadataFiles.map(async (metadataFile) => {
    // ... extract skill
  }),
);
```

**Impact**: 2-3x speedup on slow I/O (e.g., network, cold cache)

**Feasibility**: Medium

- Need careful error handling (per-skill failures don't abort)
- Maximum concurrency tuning needed
- Testing required

#### 3. Relationship Resolution Caching (saves ~100ms on second source load, MEDIUM feasibility)

**Current**: mergeMatrixWithSkills() called for each custom source load, re-resolves relationships

**Solution**: Cache by (source + ruleset) hash

```typescript
const mergeCache = new Map<string, MergedSkillsMatrix>();

function hashSourceAndRules(source: string, relationships: RelationshipDefinitions): string {
  return `${source}:${JSON.stringify(relationships).slice(0, 100)}`; // simplified
}

async function loadAndMergeFromBasePath(basePath: string): Promise<MergedSkillsMatrix> {
  const sourceConfig = await loadProjectSourceConfig(basePath);
  const hash = hashSourceAndRules(basePath, relationships);

  if (mergeCache.has(hash)) {
    return mergeCache.get(hash)!;
  }

  // ... existing merge logic
  mergeCache.set(hash, mergedMatrix);
  return mergedMatrix;
}
```

**Impact**: Saves ~100ms on repeated source loads in same session

**Feasibility**: Medium

- Cache invalidation needs thought (file changes)
- Session-scoped vs persistent cache?
- Testing required

#### 4. slug→ID Lookup Memoization (saves ~50ms per merge, LOW feasibility)

**Current**: resolveToCanonicalId() called many times during merge, does slugToId[slug] lookups

**Solution**: Already using buildSlugMap, just add memoization

```typescript
function resolveRelationships(skillId, relationships, slugMap) {
  const resolveCache = new Map<string, SkillId | null>();
  const resolve = (slug) => {
    if (!resolveCache.has(slug)) {
      resolveCache.set(slug, resolveToCanonicalId(slug, slugMap.slugToId));
    }
    return resolveCache.get(slug);
  };
  // ... use resolve()
}
```

**Impact**: ~50ms per source load (micro-optimization)

**Feasibility**: Low

- Already fast (Map lookups), minimal gains
- Cache size = unique slugs across all rules (~200)
- Good practice but not high-impact

#### 5. Build-Time Script Optimization (saves ~1.5s per build, HIGH feasibility)

**Current**: `bun run generate:types` runs on every build, reads all skills from source

**Problem**: Full file I/O + YAML parsing on every build, even if skills haven't changed

**Solution**: Cache intermediate JSON, detect skill changes

```bash
# generate-source-types.ts
const skillsSourceHash = crypto
  .createHash('md5')
  .update(JSON.stringify(extractedSkills))
  .digest('hex');

const cacheFile = '.build-cache/skills-matrix-hash.txt';
const lastHash = readFileSync(cacheFile, 'utf-8').trim();

if (skillsSourceHash === lastHash) {
  console.log('Skills unchanged, skipping matrix generation');
  exit(0);
}

// ... generate matrix
writeFileSync(cacheFile, skillsSourceHash);
```

**Impact**: ~1.5s per build (when skills unchanged)

**Feasibility**: High

- Simple file hashing
- Non-breaking change
- Easy to test
